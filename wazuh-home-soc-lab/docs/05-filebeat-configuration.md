# Filebeat Configuration

Filebeat is the piece that's easy to forget but absolutely required: it ships alerts generated by the **manager** into the **indexer**, and on its first successful start, it loads the `wazuh-alerts-*` index template that the dashboard depends on.

## Why this step exists

The manager generates alerts locally but doesn't talk to the indexer directly. Without Filebeat running, the dashboard's health check will show:
```
[Alerts index pattern] No template found for the selected index-pattern title [wazuh-alerts-*]
```

## 1. Install

```bash
sudo apt-get install filebeat
sudo curl -so /etc/filebeat/filebeat.yml https://packages.wazuh.com/4.14/tpl/wazuh/filebeat/filebeat.yml
```

## 2. Deploy certificates

Filebeat runs alongside the manager and reuses the **server node** certificate (`wazuh-1`, not a separate "filebeat" cert):

```bash
sudo mkdir -p /etc/filebeat/certs
sudo cp wazuh-certificates/wazuh-1.pem /etc/filebeat/certs/filebeat.pem
sudo cp wazuh-certificates/wazuh-1-key.pem /etc/filebeat/certs/filebeat-key.pem
sudo cp wazuh-certificates/root-ca.pem /etc/filebeat/certs/

sudo chmod 500 /etc/filebeat/certs
sudo bash -c 'chmod 400 /etc/filebeat/certs/*'
sudo chown -R root:root /etc/filebeat/certs
```

## 3. Store indexer credentials in the keystore

```bash
sudo filebeat keystore create --force
echo admin | sudo filebeat keystore add username --stdin --force
echo admin | sudo filebeat keystore add password --stdin --force
```

These populate the `${username}`/`${password}` placeholders referenced in `filebeat.yml`'s `output.elasticsearch` block.

## 4. Download the alerts index template

```bash
sudo curl -so /etc/filebeat/wazuh-template.json \
  https://raw.githubusercontent.com/wazuh/wazuh/v4.14.5/extensions/elasticsearch/7.x/wazuh-template.json
sudo chmod go+r /etc/filebeat/wazuh-template.json
```

> Use the exact version tag matching your installed Wazuh version (`v4.14.5` here) — the generic `4.x`-style alias doesn't apply to this GitHub raw content path.

## 5. Install the Wazuh module for Filebeat

```bash
sudo curl -s https://packages.wazuh.com/4.x/filebeat/wazuh-filebeat-0.5.tar.gz | sudo tar -xvz -C /usr/share/filebeat/module
```

## 6. Start the service

```bash
sudo systemctl daemon-reload
sudo systemctl enable filebeat
sudo systemctl start filebeat
```

## TLS Troubleshooting

Running `sudo filebeat test output` initially failed with:
```
handshake... ERROR x509: certificate is valid for 192.168.1.119, not 127.0.0.1
```

**Root cause:** the indexer's certificate was issued for the LAN IP (`192.168.1.119`, as set in `config.yml` during cert generation), but `filebeat.yml`'s default `output.elasticsearch.hosts` points to `127.0.0.1`. Strict TLS hostname verification checks that the address you connect to matches what's on the certificate — and `127.0.0.1` isn't on it.

**Two possible fixes:**
1. Change `filebeat.yml`'s host entry to the actual LAN IP instead of `127.0.0.1` — works, but only while the VM is actually on that IP (problematic if frequently switching between NAT/Bridged networking during setup, as this lab did).
2. **(Chosen fix)** Relax verification to validate the certificate's trust chain (still confirms it's signed by the lab's own root CA) without strictly matching the hostname:

```bash
sudo sed -i '/ssl.key:/a\  ssl.verification_mode: certificate' /etc/filebeat/filebeat.yml
sudo systemctl restart filebeat
```

This keeps the connection meaningfully secure (an attacker without the right CA-signed cert still can't impersonate the indexer) while being resilient to whichever IP/network mode the local connection happens to use — appropriate for a single-node lab where Filebeat and the indexer always run on the same machine.

Verify with:
```bash
sudo filebeat test output
```
Expected: `TLS... handshake... OK`, `talk to server... OK`, `version: 7.10.2`.
